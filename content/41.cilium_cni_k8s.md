But here's the thing — traditional CNI plugins rely heavily on **iptables** for routing, NAT, and network policies. And iptables... well, it doesn't scale well. Every Service in K8s adds more rules. Thousands of Services? Thousands of iptables rules. The lookup becomes O(n) and your damn network latency goes brrr.

Enter **Cilium** — a CNI plugin that says "Fuck iptables, we're going straight into the Kernel" using **eBPF**.

## 2. Decoding the "Core" Concepts

### What the hell is eBPF?

Think of it as writing **plugins for the Linux Kernel** — without recompiling or rebooting anything. You write a small program (bytecode), the Kernel verifies it won't crash, then hooks it directly into the networking stack. Packets get processed at the Kernel level, no userspace detour needed.

### Why does Cilium care?

Traditional CNI flow (what we did in Part 3):

```
Packet → veth → bridge/ptp → iptables (NAT, filter, mangle...) → destination
```

Cilium's flow:

```
Packet → veth → eBPF hook (TC/XDP) → destination
```

No iptables. No kube-proxy. The eBPF program handles routing, load balancing, and network policy **all inside the Kernel**. That's why Cilium is fast AF.

### Cilium replaces kube-proxy

In a standard K8s cluster, `kube-proxy` watches Services and writes iptables/IPVS rules. Cilium says "I'll handle that too" — it programs eBPF maps with Service → Pod IP mappings. So when a packet hits a Service ClusterIP, the eBPF program does the load balancing directly. No iptables chain traversal needed.

### Cilium Identity

This is Cilium's superpower. Instead of writing network policies based on IPs (which change all the time), Cilium assigns each workload a **numeric Identity** based on its labels. The eBPF program checks the identity, not the IP. Mind = blown.

## 3. Lab: Seeing Cilium in Action

### Step 1: Spin up a Kind Cluster with Cilium

We need a multi-node cluster this time, to actually see pod-to-pod communication across nodes. Also, we disable both the default CNI **and** kube-proxy (because Cilium will replace it).

Create `kind-cilium-config.yaml`:
```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
networking:
  disableDefaultCNI: true
  kubeProxyMode: none
nodes:
  - role: control-plane
  - role: worker

```

Delete old cluster (if any) and create a new one:
```bash
kind delete cluster --name kind 2>/dev/null
kind create cluster --config kind-cilium-config.yaml
```

Expected output:

### Step 3: Where Did iptables Go? — Proving eBPF Took Over

Let's look inside the Kind node to see how empty iptables is compared to a normal cluster:

```bash
docker exec -it kind-control-plane bash
```

```bash
# Check NAT table — notice how clean it is compared to a kube-proxy cluster
root@kind-control-plane:/# iptables -t nat -L -n | wc -l

# In a normal kube-proxy cluster, this would be HUNDREDS of lines.
# With Cilium, it's minimal because eBPF handles the routing.
```

Now let's see what eBPF is actually doing. Cilium stores Service → Pod mappings in eBPF maps:

```bash
# List all Services that Cilium is load-balancing via eBPF
root@kind-control-plane:/# cilium bpf lb list
```

Expected output — you should see the K8s Services (like `kubernetes` API server) mapped directly in eBPF:
```bash
SERVICE ADDRESS       BACKEND ADDRESS (REVNAT_ID) (SLOT)
10.96.0.10:53         10.244.0.100:53 (2) (1)
                      10.244.1.50:53 (2) (2)
10.96.0.1:443         172.18.0.3:6443 (1) (1)
...
```

See that? No iptables DNAT rules. The eBPF program directly maps `10.96.0.10:53` (CoreDNS Service) to the actual Pod IPs. When a packet hits the Service ClusterIP, the eBPF program rewrites the destination **inside the Kernel** — no chain traversal, no userspace bullshit.

Let's create 2 pods and trace the actual traffic:

# Create an nginx pod on any node
kubectl run nginx-server --image=nginx

curl-client    1/1     Running   0          25s   10.244.0.50   kind-control-plane

Now, let's start **Cilium Monitor** to watch packets in real-time. Open a separate terminal:
```bash
# Terminal 1: Start monitoring
cilium monitor --type trace
```

In another terminal, curl from `curl-client` to `nginx-server`:
```bash
# Terminal 2: Send traffic
NGINX_IP=$(kubectl get pod nginx-server -o jsonpath='{.status.podIP}')
kubectl exec curl-client -- curl -s -o /dev/null -w "%{http_code}" http://$NGINX_IP

```

Switch back to Terminal 1 — you should see something like:
```bash
-> endpoint 1234 flow 0x0 identity 12345->54321 state new ifindex lxcXXXX: ...
   TCP SYN
-> endpoint 1234 flow 0x0 identity 12345->54321 state established ifindex lxcXXXX: ...
   TCP SYN-ACK
```

What you're seeing is the eBPF program **tracing every single packet** as it flows through the data path. No tcpdump needed (although you can still use it). Cilium Monitor shows you the identity-based flow, which is way more useful for debugging.

### Step 5: Cilium's IPAM — How It Assigns IPs

Remember in Part 3, `host-local` IPAM stored IPs as plain text files in `/var/lib/cni/networks/`? Let's see how Cilium handles it:

```bash
# Check Cilium's IPAM mode
cilium status | grep IPAM
```

Cilium uses **Cluster Pool** IPAM by default on Kind. It stores IP allocations as Kubernetes CRDs (Custom Resource Definitions), not text files:

```bash
# See the IP pools allocated to each node
kubectl get ciliumnodes -o yaml
```

Expected output (simplified):
```yaml
apiVersion: cilium.io/v2
kind: CiliumNode
metadata:
  name: kind-worker
spec:
  ipam:
    podCIDRs:
      - 10.244.1.0/24
status:
  ipam:
    used:
      10.244.1.10:
        owner: default/nginx-server
```

See that? Instead of dumb text files on disk, Cilium tracks IP allocations as Kubernetes objects. This means:

- IPs survive node restarts (stored in etcd)
- Any component can query the API server to see which IP belongs to which Pod
- No more reading `/var/lib/cni/networks/` like a caveman

### Step 6: Identity-Based Network Policy — Cilium's Superpower

This is where Cilium truly flexes. Let's check the identities assigned to our pods:

```bash
# List all identities
cilium identity list
```

You'll see something like:
```bash
ID       LABELS
12345    k8s:app=nginx-server
54321    k8s:app=curl-client
...
```

Each pod gets an identity based on its **labels**, not its IP. Why does this matter? Because IPs change all the time (Pod restarts, rescheduling, etc.), but labels don't. Network policies based on identity are stable and fast.

Let's prove it — apply a CiliumNetworkPolicy that blocks traffic from `curl-client` to `nginx-server`:

```yaml
# deny-curl.yaml
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: deny-curl-to-nginx
spec:
  endpointSelector:
    matchLabels:
      run: nginx-server
  ingressDeny:
    - fromEndpoints:
        - matchLabels:
            run: curl-client
```

```bash
kubectl apply -f deny-curl.yaml
```

Now try to curl again:
```bash
kubectl exec curl-client -- curl -s --connect-timeout 5 http://$NGINX_IP
# Expected: Connection timed out / refused
```

Check Cilium Monitor — you'll see the packet being **DROPPED** with the identity info:
```bash
xx drop (Policy denied) flow ... identity 54321->12345 ...
```

It tells you exactly **which identity** was denied. Compare that with debugging iptables DROP rules — where you'd be grepping through hundreds of rules trying to figure out which one blocked your damn packet.

Clean up:
```bash
kubectl delete -f deny-curl.yaml
```

## 4. Summary

| Aspect | Traditional CNI (Part 3) | Cilium |
|---|---|---|
| Data plane | iptables | eBPF |
| IPAM | `host-local` (text files on disk) | CRD-based (etcd via CiliumNode) |
| Service routing | kube-proxy + iptables | eBPF LB maps |
| Network Policy | Basic (IP-based) | Identity-based + L7 aware |
| Observability | tcpdump, iptables logs | `cilium monitor`, Hubble |
| Install method | Binary in `/opt/cni/bin/` | DaemonSet + Operator |

- **Cilium** doesn't break the CNI Spec — it still uses the binary + JSON contract we explored in Part 3. But **what happens inside** that binary is completely different: instead of creating iptables rules, it loads eBPF programs into the Kernel.
- **eBPF** is the real game changer — packet processing happens inside the Kernel with O(1) hash lookups instead of iptables O(n) chain traversal.
- **Identity-based policies** make network security actually manageable. No more chasing IPs.
- There is a reason why companies like Google, AWS, and Datadog run Cilium in production. It's not hype — it's just a better way to do networking.
