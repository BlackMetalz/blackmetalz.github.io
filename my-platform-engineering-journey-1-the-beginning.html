My Platform Engineering Journey #1: The Beginning

My Platform Engineering Journey #1: The Beginning

It began with a Saturday morning coffee meeting with former colleagues. He demonstrated what he had been working on over the past three months, and I was truly amazed. He had built not just a user-friendly all-in-one portal, but specifically

  • Ticket system: Tracks progress stages and involved personnel, with notifications
  • Service section: Integrates everything internally - deployments, metrics, logs - eliminating the need for external Grafana dashboards
  • Deployment: Standard templates allowing users to get endpoints/deployments with minimal clicks

That is my motivation come from, let's begin. My first resources for this article:

  • Introduction to Platform Engineering
  • platformengineering.org
  • InternalDeveloperPlatform.org
  • Platform engineering vs DevOps

Let's talk about salary, career opportunities

  • Most of the people will tell that Platform Engineer will have higher salary than DevOps/SRE/CloudOps/Infrastructure engineers. But it will be the same result if you replace the word DevOps vs SRE/CloudOps/etc... xD. So what is the point? In my personal opinion, it depended on what you have built/able to build, not fancy certifications or beautiful resume/curriculum vitae (It is my personal opinion, I repeated xD)
  • when I'm writing this article, I don't have much idea about career opportunities, I just like keyword Platform Engineering and what it's capability, that's all nothing more. And this will consume about 6 months of mine to achieve the basic stuff I guess xD

Problems to be resolved - repeated DevOps tasks

First example - repeated tasks

I'm currently working as DevOps Engineer role. I managed GitLab groups that have hundreds repositories and 50 repositories needed to be set with CI/CD (even rewrite Dockerfile to best practices!), monitoring and so on.

When I begin to set up, first 10 repositories is completely fine, it costs about 3 hours (because they are all Go and Python). But from 11th repositories I have feeling that I'm doing repeated job and this is not the first time, before was Jenkins. So I'm starting to think is there any solution would allow me escaped from this bored task?

Second example complicated for users (Developers):

  • They need to remember which URL is for their application logs or how to query for their service.
  • Which Grafana dashboard and query is correct for their service's metrics.
  • How to check their service status, is their pod get restarted by OOM or crashed, do they know the reason if they don't have access to K8s cluster?

Third example

When I moved to new company, each of the teams have their own DevOps and config CI/CD by their own, self-managed infrastructure -> duplicate effort.

A lot of stuff I have been reading, but only thing made me feel interest. That is Platform Engineering, after watch it's concepts, I think this is the key to solve problems I see in the past (In the past not only lack of experience, I also lack of knowledge!)

What I know/understand about Platform Engineering?

  • It is based DevOps and allow DevOps scale, so it will not gonna replace DevOps. Platform Engineering evolves from DevOps!
  • Centralized developer-facing services/tools (portal is an interface in Platform Engineering)
  • Build Internal Developer Platform (IDP) as integrated product, not another internal tools or project. And user is Developer. Make developer satisfied and happy are our job!
  • An Internal Developer Platform must be built as a product following product management best practices.
  • Cover every operational necessities of entire lifecycle (services & applications).
  • Automation & Standardization, Compliance, Governance & Security by design.

Golden path for developers

From what I've learned so far... It will have project, CI/CD pipelines, document templates for developers. - Less time to ask deploy to where, who involved in that step, then the goal will be less time spend for new developer onboarding? - Everything at 1 portal, easy to remember, less effort to learn, well documented?

That is my expectation, build for developers, used by developers.

Standardization, Compliance, Governance & Security by design

Another important things I want to talk about. The problems were not every developer in our company are equal, some developers are lack of knowledge, some developers are lazy and afraid of changes. For example, I will talk about myself, I'm not only lack of basic knowledge but also lazy AF, I don't want new thing since I will have to relearn and always have thinking that using new tools will be wasted of time and that tool will send to graveyard soon xD. Hmm, seem like unrelated story to this section =.=

Lets me take some example from my knowledge: - Standardization: Standard for Dockerfile is what we want to achieve, for example with Golang, multi build layer, use Distroless as image to run application and non-root users. We will not need to config User and Group ID in securityContext of K8S pod/containers....

spec:
  template:
    spec:
      securityContext:
        seccompProfile:
          type: RuntimeDefault
      containers:
      - name: app
        image: gcr.io/distroless/static:nonroot
        securityContext:
          allowPrivilegeEscalation: false
          runAsNonRoot: true
          capabilities:
            drop:
            - ALL
          readOnlyRootFilesystem: true

And step build in Dockerfile for example with optimized flags:

# Best practice: specify Go version
FROM golang:1.21-alpine AS builder

# Best practice: use specific user ID
USER 65532:65532

RUN CGO_ENABLED=0 \
    go build \
      -ldflags="-s -w" \
      -o main .

Let me explain these flags Lets AI explain these flags:

  • CGO_ENABLED=0: Disable CGO to create a fully static binary (no C dependencies), perfect for minimal containers
  • -ldflags="-s -w": Strip debug information and symbol table to reduce binary size
  • -s: Omit the symbol table and debug information
  • -w: Omit the DWARF symbol table
  • -o main: Output binary name

With these flags, we get a smaller, more secure binary that works perfectly with Distroless images!

Most of us know how it works, how we can secure it, but we won't implement it anyway with a ton of excuse! We can not force every Golang application but if they used our IDP, this will be default, right? xD

After Standardization, after standardization? I think I already included some basic security stuff to reduce surface attack. Compliance and Governance? Developers using our product, they used what we defined!

Wait, I mentioned word "product", yes. We should treat IDP (internal developer platform, not internal developer portal haha) as a "product".

Treat IDP as a product

Why product, is it is an another internal tools?. Because developers are the main user here:

  • Mindset shift: Developers are internal customers.
  • We want to satisfy developers, make them happy by bring them good product, they only need to use our product and focus on coding/development, not environment, deployment hell. So developers are users/customers, and you know what, customers is the king xD
  • We need to understand developer pain points then solve it for them.
  • What I would approach? Implement basic stuff, tight feedback loop until we get MVP (Minimum Viable Platform).
